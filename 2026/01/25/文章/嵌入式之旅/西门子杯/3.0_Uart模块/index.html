3.0 UART模块

RossLin2026-01-25

UART 模块

1. 硬件基础

在 USB 转 TTL 模块中，CH340 是连接电脑与单片机的翻译官。

  • 核心功能：它是一颗 USB 转 UART 的协议转换芯片，能将电脑的 USB 信号转换为单片机能听懂的 TTL 串口信号。
  • 电平匹配：单片机的逻辑电平通常是 3.3V 或 5V（TTL 电平），而电脑 USB 总线电平标准完全不同。CH340 在物理层解决了这种不兼容问题。
  • 开发价值：有了它，我们才能通过电脑上的串口助手（如串口猎人、SSCOM）查看代码运行状态，实现 printf 调试。

2. 中断配置的必要性

问题：串口中断一定要配置吗？

  • 发送端：通常不配置。使用 HAL_UART_Transmit 阻塞发送即可，代码逻辑简单。虽然这会占用 CPU 时间，但在发送调试信息时通常可以接受。
  • 接收端：需要配置，否则无法接收数据
    • 如果不配置中断：CPU 必须在主循环中不断轮询（Polling）接收状态寄存器。如果 CPU 在处理其他繁重任务，串口数据就会在寄存器中被后续数据覆盖，导致数据丢失（Overrun Error）。
    • 如果配置中断：CPU 只有在硬件真正接收到数据时才会暂时停下手中的活去取货，取完立刻回来。这种异步处理方式是嵌入式实时性的基础。
  • 结论：对于超时解析法，中断是核心，否则你无法精准捕捉到每个字符到达的瞬间。

4. 串口发送函数 my_printf 详解

参数含义

  • UART_HandleTypeDef *huart：串口句柄指针。它包含了串口的所有配置信息（如波特率、引脚等）。传入指针是为了让函数知道你想用哪个串口（UART1, UART2 等）发数据。
  • const char *format：格式化字符串（如 “temp: %d”）。const 表示函数内部不会修改这个模板字符串。
  • ...：可变参数。这是 C 语言的特性，允许你传入任意数量的参数（对应模板里的 %d, %f 等）。

参数背后的逻辑：

  • UART_HandleTypeDef *huart：传递句柄指针(包含串口配置信息)，使得函数具有通用性，可以同时支持 UART1、UART2 等多个串口。
  • const char *format：这是字符串模板，包含了占位符。
  • ...：这是 C 语言的可变参数定义，配合 stdarg.h 中的宏，可以将任意数量的变量填充到模板中。

调用的 API：HAL_UART_Transmit（直接这个函数即可发送数据）

  • 函数原型：HAL_StatusTypeDef HAL_UART_Transmit(UART_HandleTypeDef *huart, uint8_t *pData, uint16_t Size, uint32_t Timeout)
  • 作用：以阻塞模式发送数据。
  • 参数说明：
    • pData：要发送的数据首地址。
    • Size：发送字节数。
    • Timeout：超时时间。如果你设为 0xFF（即 255ms），在这个时间内数据没发完，函数就会强制返回。

3. MicroLIB（微库）的取舍与报错分析

问题：为什么选择微库？如果不选会影响什么？

微库是针对嵌入式环境优化的缩减版 C 标准库。它比标准库更小，且简化了堆栈管理，最重要的是它简化了 printf 的重定向过程。

问题：为什么勾选后会报错？

报错通常是因为启动文件（startup_xxx.s）与微库的内存模型不匹配。

  • 原因：微库默认使用“两段式”存储器模型（Stack 和 Heap 分开管理）。
  • 现象：常报 __use_two_region_memory 或 __initial_sp 符号未定义。
  • 建议：如果裸机开发中标准库能正常编译且 Flash 空间充足，完全可以不勾选。如果不勾选但想用 printf，只需加入禁止半主机模式（Semihosting）的代码即可。

弱函数 (Weak Function)

  • 定义：使用 __weak 修饰的函数。
  • 作用：它提供了一个“默认实现”。如果用户在自己的代码中重新定义了一个同名函数，编译器会优先链接用户的函数；如果用户没写，则使用这个弱函数。
  • 应用：HAL 库中的所有回调函数（如 HAL_UART_RxCpltCallback）都是弱函数，方便用户按需重写。

DMA + 空闲中断

思路：利用DMA搬运工传输数据，利用空闲中断实现回调函数的功能，没有数据时通知cpu

mode有normal和circul

有何区别？

利用 DMA 搬运数据，并利用空闲中断（IDLE）通知 CPU 处理。

  • Normal 模式：传输指定长度后停止，需手动重启。
  • Circular 模式：到达缓冲区末尾后硬件自动跳回开头继续传输，无需 CPU 干预。
  • 缺陷：在停止与重启 DMA 的间隙存在数据丢失风险，且频繁的 memcpy/memset 增加了 CPU 开销。

2.处理开销： 每次中断都要进行 memcpy 和 memset，当数据包非常大或非常频繁时，CPU 会感到压力。

API 函数 功能描述
rt_ringbuffer_init 初始化环形缓冲区，设置内存池和大小。
rt_ringbuffer_put 生产者调用。向缓冲区存入数据，空间不足则丢弃超出部分。
